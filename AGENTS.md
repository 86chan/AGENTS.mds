## Core Philosophy: UNIX Way for [言語/プラットフォーム]

1. **Do One Thing and Do It Well (単一責任の徹底)**
    - [単位]は短く（理想は[行数]以内）。
    - [クラス/関数/モジュール]は一つの責務のみを持つ。
    - 複雑なロジックは小さな純粋関数に分割する。

2. **Small is Beautiful (シンプルさは正義)**
    - 複雑な継承よりもコンポジション（委譲）を選ぶ。
    - 過剰な抽象化（Over-engineering）を避け、KISS原則を守る。
    - **【ライブラリ選定】**: 基本的には標準機能で解決する。ただし、要件に対して圧倒的に優れた外部ライブラリが存在する場合は、その理由とともに提案すること。

3. **Make Every Program a Filter (データフローの重視)**
    - データは「パイプ」のように流す。[言語の関連機能]を活用し、宣言的に記述する。
    - 状態の変更（副作用）は局所化し、入力から出力を返す純粋関数的なアプローチを好む。

4. **Silence is Golden (暗黙的な失敗を許さない)**
    - エラーは握りつぶさず、明示的に処理する。
    - 型安全性・Null安全性を厳格に適用する。

## Technical Constraints & Guidelines

### Modern Practices

- **Latest Features:** 言語の最新記法を最大限に生かす。
- **Immutability First:** 状態を持たない設計を基本とする。
- **Performance:** 無駄なメモリアロケーションやリソース確保を避ける。
- **Asynchronous:** 非同期処理を正しく使用し、ブロッキング呼び出しを避ける。

### Architecture & Dependency Management

- **DI / 制御の反転**: テスト容易性が必要なコンポーネントには、依存性の注入を前提とした設計を行う。
- **疎結合**: 具体的な実装ではなく、インターフェース/型に依存させる。

### Testing

- テストが書けないコードは悪いコードである。
- ビジネスロジックは単体テストで検証可能にする。

## Documentation Rules ([言語の標準コメント形式] / Japanese / Strict)

[言語の標準コメント]生成時は以下のルールを厳守すること。

### 1. 適用範囲 (Scope)

[言語に応じた公開要素]にコメントを記述すること。
**コメント未記載のコードを検出した場合は必ず記載すること。**

### 2. 文体・形式 (Style & Format: プロダクションコード用)

- **体言止め厳守**: 要約・詳細はすべて名詞または体言止めで記述。（例：「〜の計算」「〜状態」）
- **句点なし**: 文末に句点（。）は使用しない。
- **メタ説明排除**: 「〜する関数です」「〜用の型」等の説明は排除し、事実のみを書く。

### 3. 出力例 (Examples: プロダクションコード)

#### 悪い例 (Bad - Contains Noise)

```
[言語に応じた悪い例]
説明の冗長さ、メタ説明、不要な型情報の記載
```

#### 良い例 (Good - Minimalist)

```
[言語に応じた良い例]
事実のみ、簡潔、体言止め
```

## 4. テストコードの特別規定 (Special Rules for Test Code)

テストコードは「システムの振る舞いを定義する仕様書」として機能するため、**プロダクションコードの制約（体言止め、句点なし）を除外**し、事細かに意図を明記すること。

- **目的の明文化**: 何を検証するためのテストかを明確にする。
- **Given-When-Then (AAA)**: 事前条件、実行内容、期待する結果をコメント内で明確に説明する。
- **自然な文体**: テストコードのコメントに限り、自然な文章で記述してよい。

### Code Generation Style

- **インラインコメント**: 標準コメントとは別に、複雑なロジックには「なぜそうしたか（Why）」のコメントを記述する。
- **命名**: 雄弁に。省略形は避ける。
- **完成度**: 生成するコードは、そのまま動く完全な状態にする。ビルドが通る状態を前提とする。
- **保守性**: 保守性や将来性を見越している場合はコメントを記載すること。

### Prohibited Actions

- 時代遅れの古い作法の提案。
- 巨大な「神クラス（God Object）」の作成。
- 可読性を犠牲にした過度なコードゴルフ（短縮化）。

## Testing & Coverage（テストとカバレッジ）

- **目的**: ユニット / 統合 / E2E テストを定義し、CIでカバレッジを測定して品質基準を担保する
- **カバレッジ閾値**: デフォルト 80%（プロジェクトにより引き上げ可）。CIで閾値未満なら失敗扱い
- **テスト分類**:
  - Unit: 単独コンポーネントのロジック検証
  - Integration: 複数コンポーネントの結合検証
  - E2E: 実際の外部依存を含むユーザー観点の検証
- **要求事項**:
  - すべての PR は CI のテストとカバレッジチェックをパスすること
  - 重大なロジック変更には対応するユニットテストを追加すること
  - テストは再現性があり、外部ネットワークに依存しない（必要ならモックを使用）
- **実行コマンド例**:
  - Python: python -m pip install -r requirements-dev.txt; coverage run -m pytest; coverage report --fail-under=80
  - Node: npm ci; npm test; npx nyc check-coverage --lines=80
- **CI**: .github/workflows/test.yml を用意。変更が push / PR された際に自動で検出言語（Python/Node）分岐してテスト・カバレッジを実行、coverage アーティファクトをアップロードする
